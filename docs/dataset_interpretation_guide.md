## 1. 읽기 전에 — 공통 개념

### 공간 단위

URSUS의 모든 데이터는 최종적으로 **법정동(Legal District)** 단위로 변환되어 시각화된다.

| 단위 | 설명 | 코드 형식 | 예시 |
|------|------|-----------|------|
| **행정동** (adstrd) | 행정 서비스 운영 단위. 주민센터 관할 구역 | 8자리 | `11010530` (종로구 청운효자동) |
| **법정동** (legald) | 법률상 고정된 지명 경계. 등기·공부 기준 | 8자리 (VWorld) / 10자리 (지적) | `1111010100` (종로구 청운동) |

> **왜 두 체계가 다른가?**
> 행정동은 인구 변동에 따라 통·폐합되지만, 법정동은 거의 변하지 않는다.
> 서울 열린데이터 API는 행정동 기준으로 데이터를 제공하고, VWorld WFS는 법정동 경계를 제공한다.
> URSUS는 내장된 매핑 테이블(KIKmix)로 행정동 데이터를 법정동으로 변환한다.

### 정규화(Normalization)

모든 수치 데이터는 시각화 전에 **min-max 정규화**를 거쳐 0~1 범위로 변환된다.

```
정규화값 = (원본값 - 최솟값) / (최댓값 - 최솟값)
```

- `0.0` = 분석 범위 내 최저값
- `1.0` = 분석 범위 내 최고값
- 이 값은 **상대적 순위**를 의미하며, 절대적 기준이 아님에 주의

### 가중 오버레이(Weighted Overlay)

여러 데이터셋을 동시에 선택하면 각 데이터셋의 정규화값을 **균등 평균**하여 하나의 종합 점수로 합산한다. 예를 들어 소득+인구+교통 3개를 선택하면:

```
오버레이값 = (소득_정규화 + 인구_정규화 + 교통_정규화) / 3
```

### 캐시 정책

| 항목 | 내용 |
|------|------|
| 저장 위치 | URSUS.dll과 동일 폴더 또는 지정 경로 |
| 유효 기간 | 30일 (TTL) |
| 갱신 방법 | 캐시 파일 삭제 후 컴포넌트 재실행 |
| 캐시 히트 시 응답 | 2~3초 |
| 초회 API 호출 시 응답 | ~12초 (데이터셋 크기에 따라 차이) |

### 3.1 법정동 경계 (Boundary)

#### 이 데이터는 무엇인가?

대한민국의 법정동(읍면동) 경계를 폴리곤(다각형) 지오메트리로 제공하는 **기반 레이어**이다. URSUS의 모든 데이터 시각화는 이 경계 위에 표현된다. 건축 대지가 어떤 법정동에 속하는지, 주변 법정동의 공간적 관계를 파악하는 출발점이다.

#### 출처

| 항목 | 내용 |
|------|------|
| 제공 기관 | 국토교통부 (공간정보 오픈플랫폼) |
| 서비스 | VWorld WFS (Web Feature Service) |
| API 엔드포인트 | `https://api.vworld.kr/req/wfs` |
| API 키 발급 | [vworld.kr](https://www.vworld.kr) 무료 회원가입 |
| 레이어 | `lt_c_ademd_info` (읍면동 행정구역) |

#### 데이터 형식

| 속성 | 설명 |
|------|------|
| 좌표계 | EPSG:4326 (GPS, WGS84) → URSUS 내부에서 UTM-K로 자동 변환 |
| 지오메트리 | MultiPolygon (GeoJSON) |
| 코드 | `emd_cd` (법정동 코드, 8자리) |
| 면적 | Rhino AreaMassProperties로 자동 계산 (㎡) |
| 중심점 | Centroid 자동 계산 (UTM 좌표) |

#### 건축 설계에서의 의미

- **대지 위치 확인**: 프로젝트 대지가 속한 법정동과 인접 법정동 식별
- **공간 맥락 파악**: 법정동 경계가 지형·도로·하천 등 물리적 경계와 대응되는 경우가 많아 도시 구조 이해에 도움
- **분석 기본 단위**: 이후 모든 통계 데이터가 이 경계 위에 매핑됨

#### 알려진 한계

- VWorld WFS의 경계 정밀도는 축척 1:5,000 수준으로, 필지 단위 정밀 분석에는 부적합
- 해안선·하천변 경계에서 실제 지형과 약간의 오차 존재
- 법정동 개편 시 VWorld 데이터 업데이트까지 시차 발생 가능

### 3.2 월평균 소득 (Avg Income)

#### 이 데이터는 무엇인가?

서울시 행정동별 **월평균 추정 소득**이다. 신한카드 소비 데이터를 기반으로 추정된 값으로, 해당 지역 거주자의 경제 수준을 간접적으로 나타낸다. 건축 설계에서는 대지 주변의 **상권 성격**, **임대료 수준**, **타겟 이용자 계층**을 가늠하는 데 활용한다.

#### 출처

| 항목 | 내용 |
|------|------|
| 제공 기관 | 서울특별시 (서울 열린데이터 광장) |
| 서비스명 | `VwsmAdstrdNcmCnsmpW` |
| API 엔드포인트 | `http://openapi.seoul.go.kr:8088/{키}/xml/VwsmAdstrdNcmCnsmpW/{start}/{end}/` |
| API 키 발급 | [data.seoul.go.kr](https://data.seoul.go.kr) 무료 회원가입 |
| 데이터 원천 | 신한카드 소비 패턴 기반 추정 |

#### 단위와 범위

| 항목 | 내용 |
|------|------|
| **단위** | 원(₩) / 월 |
| **사용 필드** | `MT_AVRG_INCOME_AMT` (월평균 소득금액) |
| **일반적 범위** | 약 200만 원 ~ 500만 원 |
| **시간 해상도** | 분기별 (예: `20241` = 2024년 1분기) |
| **공간 해상도** | 행정동 (서울 약 424개) |
| **총 레코드** | ~11,475행 (전체 분기 누적) |

#### URSUS 가공 방식

```
원본: 행정동 × 분기 (11,475행)
  → 전기간 단순 평균 → 행정동별 대표값 (~424개)
  → 행정동→법정동 매핑 (KIKmix)
  → min-max 정규화 [0, 1]
```

#### 건축 설계에서의 해석

| 정규화값 | 해석 | 설계 시사점 |
|----------|------|-------------|
| **0.8~1.0** | 고소득 지역 | 고급 마감, 프리미엄 상업 시설, 높은 임대료 예상 |
| **0.5~0.8** | 중상위 소득 | 중산층 대상 주거·근생 복합 적합 |
| **0.2~0.5** | 중위 소득 | 합리적 가격대 상업·주거 혼합 |
| **0.0~0.2** | 저소득 지역 | 공공임대, 사회주택, 커뮤니티 시설 고려 |

#### 주의사항

- 카드 소비 기반 **추정치**이므로 현금 거래 비율이 높은 고령 인구 밀집 지역에서 **과소 추정** 가능
- 소득 구간별 세분 데이터가 존재하나 현재는 평균값만 사용
- 서울 외 지역은 데이터 없음 (서울 열린데이터 한정)

### 3.3 상주인구 (Resident Population)

#### 이 데이터는 무엇인가?

서울시 행정동별 **주민등록 상주인구** 수이다. 해당 동에 주민등록을 둔 인구의 총수 및 연령·성별 구성을 제공한다. 건축 설계에서는 대지 주변의 **인구 밀도**, **연령 구성**, **세대 유형**(아파트/비아파트)을 파악하여 프로그램 기획에 활용한다.

#### 출처

| 항목 | 내용 |
|------|------|
| 제공 기관 | 서울특별시 (서울 열린데이터 광장) |
| 서비스명 | `VwsmAdstrdRepopW` |
| API 엔드포인트 | `http://openapi.seoul.go.kr:8088/{키}/xml/VwsmAdstrdRepopW/{start}/{end}/` |
| API 키 발급 | [data.seoul.go.kr](https://data.seoul.go.kr) 무료 회원가입 |
| 데이터 원천 | 주민등록 행정 데이터 |

#### 단위와 범위

| 항목 | 내용 |
|------|------|
| **단위** | 명 (인) |
| **사용 필드** | `TOT_REPOP_CO` (총 상주인구 수) |
| **일반적 범위** | 약 1,000명 ~ 80,000명 / 행정동 |
| **시간 해상도** | 분기별 |
| **공간 해상도** | 행정동 (서울 약 424개) |
| **총 레코드** | ~11,454행 (전체 분기 누적) |

#### 보조 필드 (현재 미사용, 향후 활용 가능)

| 필드 | 설명 | 설계 활용 가능성 |
|------|------|-----------------|
| `AGRDE_20_REPOP_CO` | 20대 인구 수 | 대학가·1인 가구 밀집도 |
| `AGRDE_60_ABOVE_REPOP_CO` | 60대 이상 인구 수 | 고령 친화 시설 수요 |
| `TOT_HSHLD_CO` | 총 세대 수 | 세대 밀도 → 주차·쓰레기 등 인프라 |
| `APT_HSHLD_CO` | 아파트 세대 수 | 주거 유형 비율 파악 |

#### URSUS 가공 방식

```
원본: 행정동 × 분기 (11,454행)
  → 전기간 단순 평균 → 행정동별 대표값 (~424개)
  → 행정동→법정동 매핑 (KIKmix)
  → min-max 정규화 [0, 1]
```

#### 건축 설계에서의 해석

| 정규화값 | 해석 | 설계 시사점 |
|----------|------|-------------|
| **0.8~1.0** | 고밀도 주거 지역 | 대규모 아파트 단지, 생활SOC 수요 높음, 주차·조경 기준 강화 |
| **0.5~0.8** | 중밀도 주거 | 다세대·다가구 밀집, 근린 생활시설 적합 |
| **0.2~0.5** | 저밀도 주거 또는 상업 혼합 | 중심 상업지구 가능성, 주간 인구 유입 확인 필요 |
| **0.0~0.2** | 극저밀도 (산지·공원·업무) | 비주거 용도 또는 개발 미진 지역 |

#### 주의사항

- 주민등록 기준이므로 **실거주 인구와 괴리** 존재 (장기 미거주 등록 세대 포함)
- 외국인 등록 인구 포함 여부 불명확
- 주간에 유입되는 통근·통학 인구는 반영되지 않음 → 생활인구와 교차 분석 권장

### 3.4 생활인구 (Living Population)

#### 이 데이터는 무엇인가?

서울시 행정동별 **실시간 체류 추정 인구**이다. KT 통신 데이터를 기반으로, 특정 시간대에 해당 동에 실제로 체류하고 있는 인구를 추정한다. 상주인구가 "어디에 사는가"를 보여준다면, 생활인구는 "어디에 있는가"를 보여준다.

#### 출처

| 항목 | 내용 |
|------|------|
| 제공 기관 | 서울특별시 (서울 열린데이터 광장) |
| 서비스명 | `SPOP_LOCAL_RESD_DONG` |
| API 엔드포인트 | `http://openapi.seoul.go.kr:8088/{키}/xml/SPOP_LOCAL_RESD_DONG/{start}/{end}/` |
| API 키 발급 | [data.seoul.go.kr](https://data.seoul.go.kr) 무료 회원가입 |
| 데이터 원천 | KT 통신 데이터 기반 추정 |

#### 단위와 범위

| 항목 | 내용 |
|------|------|
| **단위** | 명 (인) / 시간대 |
| **사용 필드** | `TOT_LVPOP_CO` (총 생활인구수) |
| **일반적 범위** | 약 500명 ~ 200,000명 / 행정동 / 시간대 |
| **시간 해상도** | 일별 × 시간대별 (00시~23시) |
| **공간 해상도** | 행정동 (서울 약 424개) |
| **총 레코드** | ~763,276행 |

#### URSUS 가공 방식

```
원본: 행정동 × 날짜 × 시간대 (763,276행)
  → 전기간·전시간대 단순 평균 → 행정동별 대표값
  → 행정동→법정동 매핑 (KIKmix)
  → min-max 정규화 [0, 1]
```

#### 건축 설계에서의 해석

| 정규화값 | 해석 | 설계 시사점 |
|----------|------|-------------|
| **0.8~1.0** | 주요 업무·상업 중심지 | 대형 오피스·상업시설, 보행 밀도 높음, 공개공지 설계 중요 |
| **0.5~0.8** | 부도심·역세권 | 복합용도 개발 적합, 유동인구 기반 상업 가능 |
| **0.2~0.5** | 주거·근린 혼합 | 생활밀착형 서비스, 커뮤니티 시설 |
| **0.0~0.2** | 주거 중심 또는 접근성 낮음 | 야간 인구 위주, 조용한 주거 환경 |

#### 상주인구 vs 생활인구 교차 분석

| 상주인구 | 생활인구 | 해석 |
|----------|----------|------|
| 높음 | 높음 | 고밀도 주거+상업 복합 (예: 강남역 일대) |
| 높음 | 낮음 | 베드타운 (예: 외곽 대규모 아파트 단지) |
| 낮음 | 높음 | 업무·상업 중심 (예: 여의도, 을지로) |
| 낮음 | 낮음 | 산지·공원·미개발 지역 |

#### 주의사항

- **현재 성능 문제로 비활성(보류) 상태** — 76만 행 이상으로 초회 API 호출 시간이 매우 김
- KT 통신 데이터 기반 추정치이며, 타 통신사 가입자 보정 방법론 비공개
- 전 시간대 평균이므로 야간 체류인구(≈상주인구)가 섞여 있음
- 주간 활동인구만 분석하려면 `TMZON_PD_SE` 09~18 필터링이 필요하나 현재 미지원

### 3.5 대중교통 승차 (Transit Boarding)

#### 이 데이터는 무엇인가?

서울시 행정동별 **일일 대중교통(버스+지하철) 총 승차 승객 수**이다. 해당 지역의 대중교통 접근성과 이용 밀도를 직접적으로 나타내며, 건축 설계에서 **교통 편의성**, **역세권 여부**, **보행 동선 계획**의 기초 자료가 된다.

#### 출처

| 항목 | 내용 |
|------|------|
| 제공 기관 | 서울특별시 (서울 열린데이터 광장) |
| 서비스명 | `tpssPassengerCnt` |
| API 엔드포인트 | `http://openapi.seoul.go.kr:8088/{키}/xml/tpssPassengerCnt/{start}/{end}/` |
| API 키 발급 | [data.seoul.go.kr](https://data.seoul.go.kr) 무료 회원가입 |
| 데이터 원천 | 교통카드 집계 데이터 |

#### 단위와 범위

| 항목 | 내용 |
|------|------|
| **단위** | 명 / 일 (일일 총 승차 승객 수) |
| **사용 필드** | `PSNG_NO` (일일 총 승차 승객 수) |
| **일반적 범위** | 약 100명 ~ 500,000명 / 행정동 / 일 |
| **시간 해상도** | 일별 (시간대별 세분 필드 존재: `PSNG_NO_00`~`PSNG_NO_23`) |
| **공간 해상도** | 행정동 (서울 약 424개) |
| **총 레코드** | ~644,687행 |

#### URSUS 가공 방식

```
원본: 행정동 × 날짜 (644,687행)
  → 전기간 단순 평균 → 행정동별 일평균 승차 승객 수
  → 행정동→법정동 매핑 (KIKmix)
  → min-max 정규화 [0, 1]
```

#### 건축 설계에서의 해석

| 정규화값 | 해석 | 설계 시사점 |
|----------|------|-------------|
| **0.8~1.0** | 주요 환승역·터미널 인접 | 역세권 개발, 지하 연결 통로, 대규모 보행 동선 계획 필수 |
| **0.5~0.8** | 주요 노선 경유 지역 | 대중교통 접근성 양호, 주차 비율 완화 근거 |
| **0.2~0.5** | 보통 수준 교통 | 표준 주차 기준 적용, 버스 노선 중심 |
| **0.0~0.2** | 교통 취약 지역 | 자가용 의존도 높음, 충분한 주차 면적 확보 필요 |

#### 주의사항

- **승차 기준**이므로 해당 동을 통과하는 이용객은 미반영 (하차 데이터 별도)
- 환승역·간선 정류장 밀집 동과 주거 중심 동 간에 **수십 배 차이** 발생 가능
- 644,687행으로 캐시 미존재 시 초회 호출 시간 상당 소요
- 버스와 지하철이 통합 집계되어 교통 수단별 분리 분석 불가

### 3.6 공시지가 (Land Price)

#### 이 데이터는 무엇인가?

국토교통부가 매년 공시하는 **표준지 공시지가**로, 토지의 단위면적(㎡)당 가격을 원 단위로 제공한다. 건축 설계에서 **사업 타당성 검토**, **토지 매입비 산정**, **지역 개발 수준 파악**의 핵심 지표이다.

#### 출처

| 항목 | 내용 |
|------|------|
| 제공 기관 | 국토교통부 (공공데이터포털) |
| 서비스명 | `ReferLandPriceService` |
| API 엔드포인트 | `http://apis.data.go.kr/1611000/nsdi/ReferLandPriceService/attr/getReferLandPriceAttr` |
| API 키 발급 | [data.go.kr](https://www.data.go.kr/data/15058747/openapi.do) 무료 즉시 발급 |
| 데이터 원천 | 국토교통부 표준지공시지가 |

#### 단위와 범위

| 항목 | 내용 |
|------|------|
| **단위** | 원(₩) / ㎡ |
| **사용 필드** | `pblntfPclnd` (공시지가) |
| **일반적 범위** | 약 10,000원/㎡ (산지) ~ 50,000,000원/㎡ (명동 등 핵심 상업지) |
| **시간 해상도** | 연 1회 공시 |
| **공간 해상도** | 법정동 (시군구 5자리 코드로 일괄 조회 → 법정동별 집계) |

#### URSUS 가공 방식

```
원본: 표준지 개별 필지
  → 시군구(5자리) 단위 일괄 조회
  → 법정동코드(ldCode, 10자리) 기준 평균 집계
  → min-max 정규화 [0, 1]
```

> 이 데이터는 행정동→법정동 변환 없이 **법정동 코드로 직접 집계**되는 유일한 데이터셋이다.

#### 건축 설계에서의 해석

| 정규화값 | 해석 | 설계 시사점 |
|----------|------|-------------|
| **0.8~1.0** | 최고가 상업·업무 지구 | 고층·고밀도 개발 경제성 확보, 지하 공간 적극 활용 |
| **0.5~0.8** | 도심 주거·상업 혼합 | 복합용도 개발 적합, 토지비 비중 높아 효율적 설계 중요 |
| **0.2~0.5** | 외곽 주거·준공업 | 토지비 부담 적정, 중밀도 개발 |
| **0.0~0.2** | 산지·녹지·개발제한 | 개발 가능성 낮거나 지가 상승 여지 높은 지역 |

#### 주의사항

- **공시지가 ≠ 실거래가** — 공시지가는 일반적으로 시세의 60~80% 수준
- 연 1회 공시로 급격한 시세 변동 반영 지연
- 별도 API 키(data.go.kr)가 필요하며, 선택적 데이터셋으로 분류됨
- 서울 외 지역에서도 사용 가능 (전국 데이터 제공)

### 3.7 용도지역 (Zoning)

#### 이 데이터는 무엇인가?

국토의 계획 및 이용에 관한 법률에 따른 **용도지역** 정보를 개발밀도 점수(1~5)로 변환한 데이터이다. 용도지역은 건축 가능한 용도, 건폐율, 용적률의 상한을 결정하는 **가장 중요한 법적 제약 조건**으로, 건축 설계의 출발점이다. URSUS는 이를 연속적 점수로 변환하여 다른 데이터셋과 함께 오버레이 분석이 가능하도록 한다.

#### 출처

| 항목 | 내용 |
|------|------|
| 제공 기관 | 국토교통부 (공공데이터포털) |
| 서비스명 | 토지이용규제정보 서비스 |
| API 엔드포인트 | `http://apis.data.go.kr/1611000/nsdi/...` |
| API 키 발급 | [data.go.kr](https://www.data.go.kr/data/15056930/openapi.do) 무료 즉시 발급 |
| 데이터 원천 | 국토교통부 토지이용규제정보 |

#### 단위와 범위

| 항목 | 내용 |
|------|------|
| **단위** | 점수 (1.0 ~ 5.0) |
| **사용 필드** | 용도지역 유형 → 개발밀도 점수 변환 |
| **일반적 범위** | 1.0 (녹지/보전) ~ 5.0 (중심상업) |
| **시간 해상도** | 비정기 (도시계획 변경 시) |
| **공간 해상도** | 법정동 (필지별 조회 → 법정동 평균 집계) |

#### 점수 체계

URSUS는 용도지역 유형을 개발밀도에 따라 1~5점 척도로 변환한다:

| 점수 | 용도지역 | 건폐율 상한 | 용적률 상한 | 성격 |
|------|----------|------------|------------|------|
| **5.0** | 중심상업지역 | 90% | 1,500% | 도심 핵심 상업 |
| **4.5** | 일반상업지역 | 80% | 1,300% | 일반 상업 활동 |
| **4.0** | 근린상업지역, 유통상업지역 | 70% | 900% | 근린 상업 |
| **3.5** | 준주거지역 | 70% | 500% | 주거+상업 혼합 |
| **3.0** | 제3종 일반주거지역 | 50% | 300% | 중밀도 주거 |
| **2.5** | 제2종 일반주거지역 | 60% | 250% | 저중밀도 주거 |
| **2.0** | 제1종 일반주거지역, 전용주거, 공업지역 | 50~60% | 150~400% | 단일 용도 |
| **1.0** | 녹지지역, 보전관리지역 | 20% | 50~100% | 개발 제한 |

#### URSUS 가공 방식

```
원본: 필지별 토지이용규제 정보
  → 법정동코드 기준 조회
  → 용도지역 유형 → 점수 변환 (1~5)
  → 법정동별 평균 점수 집계
  → min-max 정규화 [0, 1]
```

> 이 데이터는 공시지가와 마찬가지로 **법정동 코드로 직접 조회**되어 행정동→법정동 매핑이 불필요하다.

#### 건축 설계에서의 해석

| 정규화값 | 해석 | 설계 시사점 |
|----------|------|-------------|
| **0.8~1.0** | 상업·준주거 밀집 | 고층·고밀도 개발 가능, 대규모 상업·오피스 적합 |
| **0.5~0.8** | 일반주거·준주거 혼합 | 중층 주거+근생 복합, 용적률 250~500% 기대 |
| **0.2~0.5** | 전용주거·공업 | 저밀도 주거 또는 산업단지, 용도 제한 주의 |
| **0.0~0.2** | 녹지·보전 | 개발 가능성 매우 낮음, 자연환경 보전 지역 |

#### 소득·지가와의 교차 분석

| 용도지역 점수 | 지가 | 해석 |
|--------------|------|------|
| 높음 | 높음 | 활성화된 상업지구 (예: 강남역, 명동) |
| 높음 | 낮음 | 개발 잠재력 높은 지역 (규제는 허용하나 시장이 미반영) |
| 낮음 | 높음 | 규제 변경 기대 또는 희소성 기반 가치 (예: 성북동 한옥 보전) |
| 낮음 | 낮음 | 개발 부적합 지역 (산지, 녹지) |

#### 주의사항

- 점수는 URSUS가 자체 정의한 **상대적 개발밀도 척도**이며, 법적 효력이 없음
- 하나의 법정동에 복수의 용도지역이 혼재할 수 있으며, 이 경우 평균 점수가 적용됨
- 별도 API 키(data.go.kr)가 필요하며, 선택적 데이터셋으로 분류됨
- 서울 외 지역에서도 사용 가능 (전국 데이터 제공)
- 용도지역 세분류(지구·구역) 정보는 현재 반영되지 않음

## 4. 향후 확장 예정 데이터셋

아래 데이터셋은 URSUS의 '원스톱 대지 분석 도구' 비전 달성을 위해 향후 추가가 검토되는 항목이다. IDataSource 인터페이스를 통해 기존 코드 변경 없이 플러그인 방식으로 추가할 수 있도록 아키텍처가 준비되어 있다.

---

### 4.1 지적도 (Cadastral Map)

#### 개요

필지(Parcel) 경계와 지번, 지목, 면적 등을 포함하는 기본 공간 정보. 건축 허가의 기본 단위인 **필지**를 정확히 파악하기 위해 필수적이다.

#### 예상 출처

| 항목 | 내용 |
|------|------|
| 후보 API | 국가공간정보포털 (nsdi.go.kr) 연속지적도 WFS |
| 대안 | VWorld 지적도 레이어 (`lt_c_lhblot`) |
| 좌표계 | EPSG:4326 또는 EPSG:5186 (중부원점TM) |

#### 핵심 속성

| 속성 | 설명 | 단위 |
|------|------|------|
| 지번 | 대지의 법적 주소 번호 | 번지 |
| 지목 | 토지의 법적 용도 분류 (대·전·답·임야 등 28종) | 코드 |
| 면적 | 공부상 토지 면적 | ㎡ |
| 소유 구분 | 국유·공유·사유 | 코드 |

#### 건축 설계 활용

- 정확한 대지 경계 확인 (건축선·인접도로 관계 파악)
- 합필·분필 가능성 검토
- 지목 확인을 통한 개발 가능 여부 사전 판단

---

### 4.2 경사도 (Slope)

#### 개요

대지의 **지형 경사 정도**를 나타내는 지표. 건축 설계에서 절·성토 규모, 접근 동선, 구조 방식을 결정하는 핵심 물리적 조건이다.

#### 예상 출처

| 항목 | 내용 |
|------|------|
| 후보 데이터 | 국토지리정보원 수치표고모델(DEM) |
| 대안 | 환경부 경사도 분석 자료, OpenTopography SRTM |
| 해상도 | 5m~30m 격자 |
| 형식 | GeoTIFF 래스터 또는 파생 WMS 서비스 |

#### 핵심 속성

| 속성 | 설명 | 단위 |
|------|------|------|
| 표고 | 해발 높이 | m |
| 경사도 | 수평 거리 대비 높이 변화 | 도(°) 또는 % |
| 경사 방향 | 가장 급한 내리막 방향 | 방위각 (0°~360°) |

#### 건축 설계 활용

| 경사도 | 해석 | 설계 시사점 |
|--------|------|-------------|
| **0°~5°** | 평탄지 | 일반적 건축에 적합, 최소 절·성토 |
| **5°~15°** | 완경사 | 단차 설계, 필로티·스킵플로어 고려 |
| **15°~25°** | 급경사 | 옹벽·구조 보강 필요, 테라스형 배치 |
| **25° 이상** | 급경사지 | 개발 부적합 또는 특수 구조 요구, 재해 위험 검토 |

### 4.4 일조 분석 (Solar Access)

#### 개요

대지에 도달하는 **태양 복사 에너지량** 및 **일조 시간**을 분석하는 데이터. 건축법상 일조권 확보 기준 검토, 에너지 효율 설계, 옥외 공간 쾌적성 판단에 활용한다.

#### 예상 출처

| 항목 | 내용 |
|------|------|
| 후보 데이터 | 기상청 일사량 관측 데이터 (data.go.kr) |
| 대안 | NASA POWER API (전지구), Ladybug Tools EPW 파일 |
| 해상도 | 관측소 기반 점 데이터 → 보간 필요 |

#### 핵심 속성

| 속성 | 설명 | 단위 |
|------|------|------|
| 수평면 전일사량 | 단위면적에 도달하는 총 태양 에너지 | kWh/㎡/일 |
| 일조 시간 | 직달 일사가 도달하는 시간 | 시간/일 |
| 동지 일조 시간 | 일조권 검토 기준 (연중 최단일) | 시간 |

#### 건축 설계 활용

- **건축법 일조권 기준**: 동지일 기준 연속 2시간 또는 통산 4시간 확보 (공동주택)
- 남측 개구부 계획, 차양 설계의 근거
- 태양광 패널 설치 잠재력 평가
- 외부 공간(마당, 옥상정원) 쾌적성 예측

### 4.5 건축물 현황 (Building Registry)

#### 개요

대지 주변의 **기존 건축물 정보**(용도, 층수, 연면적, 구조, 준공일 등). 도시 맥락 파악과 스카이라인 분석, 인접 건물과의 관계 설정에 활용한다.

#### 예상 출처

| 항목 | 내용 |
|------|------|
| 후보 API | 국토교통부 건축물대장 정보 서비스 (data.go.kr) |
| 대안 | 서울시 건축물현황 정보, VWorld 건물통합정보 3D |
| 서비스 유형 | REST API (필지/PNU 코드 기반 조회) |

#### 핵심 속성

| 속성 | 설명 | 단위 |
|------|------|------|
| 주용도 | 건물의 주 사용 목적 | 코드 (근생, 공동주택 등) |
| 층수 | 지상/지하 층수 | 층 |
| 연면적 | 각 층 바닥면적의 합계 | ㎡ |
| 건축면적 | 건물이 차지하는 대지 면적 | ㎡ |
| 높이 | 건물의 최고 높이 | m |
| 준공일 | 건물 완공 시점 | 날짜 |

#### 건축 설계 활용

- 인접 건물 높이·용도를 통한 스카이라인 맥락 파악
- 일조·조망 침해 사전 검토
- 주변 건물 노후도(준공일)를 통한 재개발 가능성 판단
- 지역 내 건물 용도 비율 파악 → 프로그램 기획 근거

### 4.6 도로망 (Road Network)

#### 개요

대지에 접하는 **도로의 위치, 폭원, 등급** 정보. 건축법상 도로 접도 의무, 건축선 후퇴, 차량 진출입 계획의 기본 데이터이다.

#### 예상 출처

| 항목 | 내용 |
|------|------|
| 후보 API | VWorld 도로명주소 도로 레이어, 국가교통DB |
| 대안 | OpenStreetMap 도로 데이터 |
| 형식 | WFS (라인 지오메트리) |

#### 핵심 속성

| 속성 | 설명 | 단위 |
|------|------|------|
| 도로폭 | 도로의 실측 또는 공부상 폭원 | m |
| 도로 등급 | 고속도로/일반국도/시도/구도/이면도로 | 등급 코드 |
| 차선 수 | 편도 차선 수 | 개 |

#### 건축 설계 활용

| 도로폭 | 건축법 적용 | 설계 시사점 |
|--------|------------|-------------|
| **4m 미만** | 건축선 후퇴 (도로 중심 2m) | 대지 면적 감소, 1층 셋백 |
| **4~8m** | 일반 접도 의무 충족 | 표준 건축 가능 |
| **8~12m** | 소방차 진입 가능 | 중규모 건축 적합 |
| **12m 이상** | 대로변 | 상업 시설 적합, 소음·분진 차단 설계 |

## 5. 정규화와 오버레이 이해하기

### 정규화가 필요한 이유

각 데이터셋의 원본 단위와 범위가 전혀 다르다:

| 데이터셋 | 단위 | 일반적 범위 |
|----------|------|-------------|
| 소득 | 원/월 | 200만~500만 |
| 인구 | 명 | 1,000~80,000 |
| 교통 | 명/일 | 100~500,000 |
| 지가 | 원/㎡ | 10,000~50,000,000 |

이 값들을 직접 합산하면 단위가 큰 데이터(지가)가 결과를 지배하게 된다. min-max 정규화로 모든 값을 0~1 범위로 변환하면 각 데이터셋이 동등한 가중치로 비교·합산된다.

### 오버레이 해석 예시

**시나리오**: 상업시설 대지를 찾고 있다 → 소득 + 교통 + 생활인구 선택

| 법정동 | 소득(정규화) | 교통(정규화) | 생활인구(정규화) | 오버레이 | 해석 |
|--------|------------|------------|--------------|---------|------|
| 강남구 역삼1동 | 0.85 | 0.92 | 0.88 | **0.88** | 고소득+고교통+고유동 → 최적 |
| 도봉구 방학1동 | 0.30 | 0.25 | 0.35 | **0.30** | 주거 중심, 상업 부적합 |
| 마포구 서교동 | 0.55 | 0.65 | 0.78 | **0.66** | 중상위, 소규모 상업 적합 |

> 오버레이값은 **"선택한 기준에 대한 상대적 적합도"**를 의미한다. 절대적 우열이 아닌 비교 도구로 활용해야 한다.

## 6. 데이터 해석 시 주의사항

### 공통 주의사항

1. **상대값 vs 절대값**: 정규화된 0~1 값은 분석 범위(현재 서울) 내 상대 순위이다. 범위가 바뀌면 같은 지역도 다른 값을 가진다.

2. **시간 지연**: 데이터의 최신성은 출처 기관의 업데이트 주기에 의존한다. 분기별 데이터는 최대 3개월 전 상황을 반영한다.

3. **공간 해상도 한계**: 법정동 단위 집계는 동 내부의 편차를 감추므로, 특정 필지의 미시적 특성은 별도 현장 조사가 필요하다.

4. **추정치 데이터**: 소득(카드 소비 기반)과 생활인구(통신 데이터 기반)는 모두 추정치이다. 정책 결정이 아닌 설계 초기 판단의 참고 자료로 활용해야 한다.

5. **매핑 오차**: 행정동→법정동 매핑 시 1:N 관계에서 평균값이 적용되므로, 매핑 미존재 법정동에는 전체 평균이 할당된다.

### 데이터 신뢰도 등급

| 데이터셋 | 신뢰도 | 근거 |
|----------|--------|------|
| 법정동 경계 | ★★★★★ | 공적 지오메트리, 법적 효력 |
| 상주인구 | ★★★★☆ | 행정 등록 확정 수치 (실거주 괴리 존재) |
| 대중교통 승차 | ★★★★☆ | 교통카드 실측 집계 |
| 공시지가 | ★★★★☆ | 공적 감정 평가 (시세와 괴리 존재) |
| 월평균 소득 | ★★★☆☆ | 카드 소비 기반 추정치 |
| 생활인구 | ★★★☆☆ | 통신 데이터 기반 추정치 |

## 7. 건축 설계 단계별 활용 시나리오

### 시나리오 1: 대지 선정 (Site Selection)

> "여러 후보지 중 어디가 가장 적합한가?"

**권장 데이터 조합**: 전체 데이터셋 오버레이

1. 모든 활성 데이터셋을 선택하여 종합 오버레이 실행
2. 히트맵에서 프로젝트 목적에 맞는 높은/낮은 값 지역 식별
3. 후보 법정동 목록 CSV 내보내기 후 비교표 작성

### 시나리오 2: 대지 분석 (Site Analysis)

> "선정된 대지의 특성을 파악하고 싶다."

**권장 접근**: 개별 데이터셋을 하나씩 확인

1. 법정동 경계로 대지 위치 확인
2. 소득 데이터 → 주변 경제 수준 파악
3. 상주인구 → 거주 밀도와 연령 구성 추정
4. 대중교통 → 접근성 평가
5. 공시지가 → 토지 가치와 개발 밀도 기대값
6. 각 레이어별 시각화를 캡처하여 분석 보고서 구성

### 시나리오 3: 프로그램 기획 (Programming)

> "이 대지에 어떤 용도가 적합한가?"

**권장 데이터 조합**: 소득 + 생활인구(또는 상주인구) + 교통

| 분석 결과 | 제안 프로그램 |
|-----------|--------------|
| 고소득 + 고유동 + 고교통 | 프리미엄 상업·오피스 |
| 중소득 + 고상주 + 중교통 | 근린 생활시설 + 주거 복합 |
| 저소득 + 고상주 + 저교통 | 공공시설, 커뮤니티 센터, 사회주택 |
| 고지가 + 저상주 + 고교통 | 업무시설, 호텔, 판매시설 |

### 시나리오 4: 스튜디오 과제 (학생용)

> "교수님이 대지 분석 발표를 요구했다."

**15분 워크플로**:

1. URSUS Solver 컴포넌트 배치 (입력 없이 바로 실행)
2. Visualizer 연결 → 전체 오버레이 히트맵 캡처
3. DataSet을 하나씩 바꿔가며 개별 레이어 캡처 (소득 → 인구 → 교통)
4. CSV 내보내기로 수치 데이터 확보
5. 캡처 이미지 + 수치 데이터로 분석 슬라이드 구성
